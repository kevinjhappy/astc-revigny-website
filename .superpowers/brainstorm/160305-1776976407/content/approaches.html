Architecture — 3 approches DDD avec Symfony

Les décisions stack sont prises. La question est : comment structurer le DDD ?

src/
Tournament/
Domain/ Entity/ Repository/ Service/
Application/ Command/ Query/ Handler/
Infrastructure/ Doctrine/ Controller/
Member/
Domain/ Application/ Infrastructure/
Registration/
Domain/ Application/ Infrastructure/

A — DDD Bounded Contexts complets ✅ Recommandé

3 contextes isolés : Tournament, Member, Registration. Chaque contexte a son Domain, Application (CQRS léger), et Infrastructure. Communication par événements Symfony.

Pros

  • SOLID respecté
  • Testabilité maximale
  • Évolutif

Cons

  • Plus verbeux au départ
src/
Entity/ Tournament.php Member.php
Repository/ TournamentRepo.php
Service/ TournamentService.php
Controller/ PublicController.php
Controller/ AdminController.php
Form/ EventSubscriber/ ...

B — Structure Symfony classique

Organisation par type technique (Entity, Repository, Service, Controller). Moins de fichiers, plus rapide à démarrer. Pas de séparation Domain/Application/Infrastructure.

Pros

  • Familier Symfony
  • Moins de fichiers

Cons

  • Difficile à faire évoluer
  • Moins SOLID
src/
Tournament/ Member/ Registration/
Domain/ (Entity, VO, Events)
Application/ (UseCase, DTO)
Infrastructure/ (Doctrine, Ctrl)
Shared/
ValueObjects/ DomainEvent/ Bus/

C — DDD + Shared Kernel + Event Bus

Comme A mais avec un Shared Kernel pour les Value Objects communs (PhoneNumber, Email) et un Event Bus explicite entre contextes. Full Hexagonal Architecture.

Pros

  • Architecture la plus propre
  • DDD strict

Cons

  • Complexité élevée
  • Overkill pour ce périmètre
